## 2주차(process) 면접 질문

💡 질문 1.

시스템 콜과 인터럽트의 차이는 무엇이며, 시스템 콜이 왜 인터럽트를 이용하는가?

핵심 포인트
	•	시스템 콜: 사용자 프로그램이 커널 서비스를 요청하는 “소프트웨어 인터럽트”
	•	인터럽트: 하드웨어나 커널이 CPU 실행을 중단시키는 메커니즘
	•	시스템 콜은 유저 모드 → 커널 모드로 안전하게 진입하기 위해 인터럽트를 사용

백엔드 연결
	•	DB I/O, 파일 I/O 모두 시스템 콜 기반 → blocking I/O의 근본 이해
모바일 연결
	•	안드(iOS)에서 네이티브 호출이나 파일 접근 시 시스템 콜 경로 동일

⸻

💡 질문 2.

시스템 콜은 커널 모드 전환인데, 왜 컨텍스트 스위치보다 훨씬 가볍다고 할 수 있을까요?

핵심 포인트
	•	시스템 콜은 같은 프로세스 내에서 커널 모드로 잠깐 진입 → mode switch
	•	컨텍스트 스위치는 다른 프로세스로 넘어감 → TLB flush, 캐시 무효화, PCB 교체
	•	비용 차이 원인: 주소 공간 전환, 캐시 손실, 스케줄러 개입

백엔드 연결
	•	네트워크 서버에서 syscall-heavy 작업 최적화, epoll/select 이해
모바일 연결
	•	안드·iOS 모두 시스템 콜 빈도 최소화가 성능에 직접 영향 (예: 파일·소켓 I/O batching)

⸻

💡 질문 3.

프로세스 간 통신(IPC)은 왜 시스템 콜을 필요로 하고, Shared Memory와 Message Passing의 차이는 무엇인가요?

핵심 포인트
	•	IPC는 커널이 관리하는 자원(메모리, 큐 등)을 공유하기 때문에 시스템 콜이 필요
	•	Shared Memory: 빠르지만 직접 동기화 필요
	•	Message Passing: 느리지만 커널이 동기화 보장

백엔드 연결
	•	마이크로서비스 간 통신(RPC, Socket)이 OS IPC의 확장 개념
모바일 연결
	•	안드: Binder, iOS: XPC → 모두 message passing 기반 IPC 구현체
