# MySQL(InnoDB)에서의 데드락 처리

## 데드락 처리 방법

> 예방/회피보다는 감지해서 롤백하는 방향으로 진행
>

1. 대기 관계에 있는 트랜잭션(스레드라고 보면 됨.) 간의 사이클을 감지해서 데드락이 발생했는지를 판단한다.
2. 데드락이 감지되면 희생자 트랜잭션을 하나 선택해서 롤백하고 다른 트랜잭션이 진행되게 된다.
3. 중단된 트랜잭션에 대해서는 쿼리를 실행한 서버에서 실패에 대한 에러를 받게 됨. → 서버 코드를 통해 트랜잭션을 재시도하도록 구현 필요

## 이 방법을 쓰는 이유

DB는 동시에 처리되는 트랜잭션이 많아서 어떤 요청을 미리 거절해서 안전 상태만 유지 하는 건 현실적으로 어렵다. → 그래서 일단 동시성을 최대한 허용하고 정말로 막히는 순간만 빠르게 찾아서 끊는 전략을 쓴다.

## 실무상에서 데드락 해결하기 위한 방법

1. 항상 같은 순서로 row 잠그기

   ex.) 여러 테이블이나 row 업데이트 시 정렬된 순서로 처리

2. 트랜잭션을 짧게 한다.

   트랜잭션 안에서 외부 API 호출이나 느린 작업, 사용자 입력 대기 같은 것을 절대 추가하지 않는다.

3. 필요한 인덱스 제대로 설정한다.

   인덱스가 없으면 의도한 것과는 다르게 더 큰 범위로 락 경합이 발생할 수 있다.

4. 재시도 로직

   데드락, 락 타임아웃 같은 에러를 반환했을 때 재시도하는 로직을 추가한다.


## 서버 단에서의 데드락 처리 방법

1. 위에서 말했듯이 반환된 에러를 확인해 재시도 로직 추가
2. Redis 등을 사용해 분산락을 사용
    - 여러 서버에서 사용하는 하나의 Lock을 사용해서 자원을 사용할 수 있을 때까지 무한 대기 시킨다.
    - 현실적으로는 무한 대기를 막기 위해 Lock에 만료 기한을 추가한다.
3. 애플리케이션 레벨의 데드락을 사용한다.
    - 뮤텍스나 큐를 사용해서 논리적으로 Lock을 정의해서 사용한다.
