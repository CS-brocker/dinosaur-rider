# 가상 메모리

## 가상 메모리란?

**가상 메모리(Virtual Memory)** 는 프로세스가 실제 물리 메모리(RAM)의 크기나 위치에 얽매이지 않고, 자신만의 연속적이고 거대한 메모리 공간(=논리 주소 공간)을 사용하는 것처럼 보이게 해주는 기법이다.

쉽게 말하면, **"프로세스가 보는 메모리"와 "실제 RAM"을 분리**해서, 그 사이를 운영체제와 하드웨어(MMU)가 알아서 매핑해 주는 구조다. 프로세스는 자기가 RAM 어디에 올라가 있는지, 심지어 지금 RAM에 올라가 있긴 한지조차 신경 쓸 필요가 없다.

## 가상 메모리를 사용해서 얻는 이점

### 1. 논리 메모리와 물리 메모리의 분리
프로세스는 자신만의 깔끔한 주소 공간(예: 0번지부터 시작하는 연속된 공간)을 본다. 실제로는 물리 메모리 여기저기 흩어져 있어도, 프로그래머와 프로세스 입장에서는 신경 쓸 필요가 없다. 덕분에 **프로그래밍이 단순해지고**, OS는 물리 메모리를 유연하게 관리할 수 있다.

### 2. 물리 메모리보다 큰 프로세스 실행 가능
**요구 페이징(demand paging)** 덕분에, 프로세스 전체를 한 번에 메모리에 올릴 필요가 없다. 당장 실행에 필요한 페이지만 RAM에 올리고, 나머지는 디스크(스왑 공간)에 두었다가 필요할 때 가져온다.

> ※ 동적 적재(dynamic loading)와 혼동하기 쉽다. 동적 적재는 **프로그래머가 직접** 루틴 단위로 적재 시점을 관리하는 기법이고, 가상 메모리의 요구 페이징은 **OS가 자동으로** 페이지 단위로 처리해 준다.
>
> 예를 들어 C 언어에서는 `dlopen()`, `dlsym()`(Unix/Linux)이나 `LoadLibrary()`(Windows) 같은 API를 **프로그래머가 직접 호출**해서 "이 루틴을 지금 메모리에 올려라"라고 명시해 줘야 했다. 더 옛날에는 **오버레이(overlay)** 구조를 직접 설계해서 같은 메모리 영역에 서로 다른 모듈을 번갈아 올리기도 했다. 즉 동적 적재는 어디까지나 **사용자 수준의 수동 기법**이다.
>
> 반면 가상 메모리는 프로그래머가 신경 쓰지 않아도 OS가 알아서 처리해 주므로, "물리 메모리보다 큰 프로그램 실행"은 본질적으로 **가상 메모리가 제공하는 이점**이라고 보는 게 맞다.

### 3. 페이지 공유를 통한 프로세스 간 메모리 공유
서로 다른 프로세스가 동일한 물리 페이지를 자신의 논리 주소 공간에 매핑할 수 있다. 이를 통해
- 공유 라이브러리(예: libc)를 여러 프로세스가 메모리에 한 벌만 올려놓고 같이 쓰고,
- 프로세스 간 통신(IPC)에 공유 메모리를 활용할 수 있다.

메모리 절약과 효율적인 협업이 동시에 가능해진다.

### 4. 프로세스 생성(fork) 속도 향상
`fork()`로 자식 프로세스를 만들 때, 부모의 메모리 전체를 복사하지 않고 **페이지 테이블만 복사**한 뒤 같은 물리 페이지를 가리키게 한다. 이후 어느 한쪽이 그 페이지에 쓰기를 시도하는 순간에만 페이지를 복제한다(**Copy-on-Write, COW**).

대부분의 페이지는 읽기만 하거나, 자식이 곧바로 `exec()`을 호출해 새 프로그램으로 갈아치우기 때문에, **실제로 복사되는 페이지는 극히 일부**다. 결과적으로 프로세스 생성 비용이 크게 줄어든다.


### 잠깐, 페이징이랑 가상 메모리는 결국 같은 거 아닌가? (AI Q&A)

위에서 나열한 이점 중 일부가 사실은 이전 장에서 다룬 **페이징(paging)** 자체의 이점처럼 보일 수 있다. 헷갈리는 부분을 AI에게 물어본 내용을 정리했다.

**Q.** 페이징만으로도 프로세스를 쪼개서 메모리에 올리는 것도 되고, 논리 주소를 물리 주소로 매핑하는 것도 다 되잖아? 그럼 "논리 메모리와 물리 메모리의 분리"나 "물리 메모리보다 큰 프로세스 실행" 같은 건 사실 가상 메모리만의 이점이 아니라 페이징의 이점 아닌가? 어차피 MMU(메모리 관리 장치)가 알아서 매핑해 주는 거고, 가상 메모리랑은 직접적인 상관이 없는 것 같은데. "우리가 신경 안 써도 된다"는 점이 가상 메모리의 핵심인 건가?

**A.** 직관의 절반은 정확하다. 하나씩 짚어보면:

- **"논리 메모리와 물리 메모리의 분리"** → 이건 엄밀히 말하면 **페이징의 이점**이다. 페이지 테이블과 MMU만 있어도 이미 달성된다. 책에서 가상 메모리의 이점으로 같이 묶어 적는 경우가 많은데, 가상 메모리는 그 위에서 분리 효과를 더 극단적으로 활용하는 것에 가깝다.

- **"물리 메모리보다 큰 프로세스 실행"** → 이건 페이징만으로는 **불가능**하다. 순수 페이징은 여전히 *프로세스의 모든 페이지가 RAM에 올라가 있어야* 실행할 수 있기 때문이다. 100MB짜리 프로세스를 50MB RAM에서 돌리려면, 디스크(스왑)와 페이지 폴트 처리 메커니즘이 필요하고, 이게 바로 가상 메모리가 새로 도입하는 영역이다.

즉, **페이징은 "RAM 안에서 흩어진 페이지를 모아 보여주는" 기법**이라면, **가상 메모리는 그 페이지 일부를 디스크에까지 확장한 기법**이라고 보면 깔끔하다.

"우리가 신경 안 써도 된다"는 표현도 사실 정확하다. 프로그래머/프로세스 입장에서 페이지 폴트도, 스왑도, 교체도 보이지 않게 OS가 처리해 주기 때문에, 결과적으로는 "그냥 메모리가 큰 것처럼 보인다"가 가상 메모리의 사용자 측 효과다.

> **비유.** 페이징은 "책장에 책을 흩어 놔도, 카탈로그(페이지 테이블)로 책 위치를 찾을 수 있다"는 것이고, 가상 메모리는 "책장 용량보다 책이 많아도, 자주 안 보는 건 창고(디스크)에 두었다가 필요할 때 꺼내온다"는 것. 창고와 꺼내오는 로직이 가상 메모리의 정체.

## 요구 페이징

**요구 페이징(demand paging)** 은 말 그대로 "**요구(접근)될 때** 페이지를 메모리에 올리는" 방식이다. 프로세스를 시작할 때 모든 페이지를 미리 메모리에 적재하지 않고, 실제로 그 페이지가 참조되는 순간에 비로소 디스크에서 가져온다.

장점은 명확하다.
- 프로세스 시작이 빠르다 (전부 적재할 필요 없음)
- 메모리를 아낄 수 있다 (안 쓰는 페이지는 안 올림)
- 더 많은 프로세스를 동시에 돌릴 수 있다

### 유효/무효 비트의 의미 확장

이 비트는 새로운 게 아니다. **9장 페이징에서 등장했던 그 비트를 그대로 재사용**하되, 의미만 한 단계 더 확장한다.

| | 9장 (페이징) | 10장 (가상 메모리 / 요구 페이징) |
|---|---|---|
| **유효(valid)** | 이 프로세스의 합법적 주소 공간에 속함 | 합법적 주소 공간에 속함 **+ 현재 RAM에 적재되어 있음** |
| **무효(invalid)** | 프로세스 주소 공간 밖 (불법 접근) | (a) 주소 공간 밖이거나, (b) 합법적이지만 **지금은 디스크에 있음** |

비트는 1개로 그대로지만, 무효의 의미가 (a) "쓰면 안 되는 페이지"와 (b) "써도 되는데 아직 메모리에 안 올라온 페이지" 둘로 늘어난 셈이다. 둘의 구분은 무효 트랩이 발생한 뒤 OS가 PCB의 합법 주소 범위 등을 참고해서 판단한다.

- (a)면 → **프로세스 중단(Segmentation Fault)**
- (b)면 → **페이지 폴트(page fault)** 처리

### 페이지 폴트 트랩 처리 순서

1. 무효로 표시된 페이지에 접근 → MMU가 트랩 발생
2. OS가 트랩을 받아 합법적 참조인지 확인
3. 빈 프레임을 찾는다 (없으면 페이지 교체 알고리즘으로 하나 비움)
4. 디스크(스왑 공간)에서 해당 페이지를 빈 프레임으로 읽어옴
5. 페이지 테이블 갱신, 유효 비트 set
6. **트랩을 일으킨 명령어를 처음부터 다시 실행(restart)**

프로세스 입장에서는 잠깐 멈췄다 다시 실행될 뿐, 폴트가 일어났는지조차 모른다.

### 순수 요구 페이징 (Pure Demand Paging)

극단적으로는 **처음에 단 한 페이지도 미리 안 올리는** 방식도 가능하다. 프로세스가 첫 명령어를 fetch하려는 순간부터 페이지 폴트가 터지고, 거기서부터 필요한 페이지가 하나씩 올라온다. 이론적으로 메모리를 가장 아끼지만, 시작 직후 폴트가 폭주해서 비효율적이라 실제로는 잘 쓰이지 않는다.

### 그런데 왜 폴트가 매번 안 터지나? — 참조의 지역성

"매번 페이지 폴트가 나면 디스크 I/O 때문에 엄청 느릴 텐데?" 라는 의문이 들 수 있다. 하지만 실제로는 페이지 폴트가 **드물게만** 발생한다. 그 이유가 **참조의 지역성(locality of reference)** 이다.

- **시간적 지역성**: 최근에 접근한 메모리는 곧 다시 접근될 가능성이 높다 (반복문, 자주 호출되는 함수)
- **공간적 지역성**: 어떤 주소에 접근하면 근처 주소도 곧 접근된다 (배열 순회, 순차적 코드 실행)

→ 한 번 메모리에 올라온 페이지가 한동안 계속 쓰이기 때문에, 폴트는 가끔만 일어난다. 그래서 요구 페이징이 **실용적으로 잘 작동**하는 것이다. (지역성이 깨지는 워크로드는 폴트가 폭주하는데, 이걸 **스래싱(thrashing)** 이라고 부르고 뒷장에서 따로 다룬다.)

### 명령어 재시작 문제 — 메모리를 많이 건드리는 명령어의 함정

페이지 폴트가 나면 OS는 트랩을 일으킨 명령어를 **처음부터 다시 실행**한다. 보통은 문제없지만, **한 명령어가 여러 메모리 위치를 건드리는 경우**가 까다롭다.

예를 들어 어떤 CPU의 **블록 이동 명령**(예: IBM의 MVC — 256바이트 메모리 블록을 src에서 dst로 복사):
- 100바이트 복사한 시점에 dst의 다음 페이지에서 폴트 발생
- 페이지 가져온 뒤 명령어를 "처음부터" 재실행
- 그런데 **src와 dst가 겹쳐 있다면**? 이미 덮어쓴 영역의 값이 달라져 있어서, 재실행하면 결과가 망가진다.

3-주소 명령어(예: `ADD R1, R2, R3` 형태로 여러 피연산자를 읽고 결과를 또 다른 위치에 쓰는 명령)도 비슷한 위험이 있다.

**해결책 (CPU/OS가 함께 처리):**
1. **선검사(pre-check)**: 명령어 시작 전, 양쪽 페이지를 미리 건드려서 폴트가 날 거면 *먼저* 일으켜 둠 → 본 실행 중에는 폴트 안 남
2. **임시 레지스터 백업**: 덮어쓰기 전 원본을 백업해두고, 폴트 발생 시 롤백
3. **재시작 가능 명령어(restartable instruction)** 설계: 명령어를 항상 처음부터 안전하게 재실행해도 결과가 같도록 하드웨어 차원에서 보장

**시사점:** 가상 메모리를 지원하는 CPU는 이런 메커니즘을 반드시 갖춰야 한다. 그래서 **메모리를 많이 건드리는 명령어(블록 이동, 문자열 처리 등)는 설계할 때 페이지 폴트 발생 가능성을 항상 고려**해야 한다.

### 가용 프레임 리스트 (free-frame list)

페이지 폴트가 발생하면 OS는 디스크에서 페이지를 읽어와 **어딘가 빈 프레임**에 넣어야 한다. 이때 "빈 프레임이 어디 있는지" 매번 메모리를 훑어 찾는 건 비효율이라, OS는 미리 **사용 가능한(비어 있는) 프레임 목록을 연결 리스트로 관리**해 둔다. 이게 가용 프레임 리스트다.

- 페이지 폴트 발생 → 리스트에서 한 프레임을 떼어서 사용
- 프로세스가 종료되거나 페이지가 교체되어 비워지면 → 다시 리스트에 반환
- 부팅 시점에는 거의 모든 프레임이 이 리스트에 들어있고, 시간이 지나면서 사용/반환을 반복

### Zero-Fill-on-Demand

가용 프레임 리스트에서 프레임 하나를 떼어서 프로세스에게 주기 전에, OS는 그 프레임의 내용을 **모두 0으로 초기화**한다. 이를 **zero-fill-on-demand**라고 부른다.

왜 굳이 0으로 채울까? → **보안 때문이다.**
- 빈 프레임이라고는 하지만, 그 안에는 **이전에 그 자리를 썼던 다른 프로세스의 데이터(비밀번호, 토큰, 키 등)가 그대로 남아있을** 수 있다.
- 만약 0으로 초기화하지 않고 그대로 새 프로세스에게 넘기면, 새 프로세스는 자기 메모리를 읽는 척하면서 **이전 프로세스의 잔여 데이터를 훔쳐볼 수 있다.** 이건 명백한 정보 누출 취약점이다.
- 그래서 OS는 프레임을 넘기기 전에 0으로 싹 밀어버려서, 다른 프로세스의 흔적이 남지 않도록 보장한다.

이름이 "on-demand"인 이유는, 미리 모든 빈 프레임을 0으로 채워두는 게 아니라 **실제로 프로세스에게 줄 때(=요구 시점에)** 그 프레임만 0으로 초기화하기 때문이다.

### 요구 페이징 성능
책에서 페이지 폴트 과정을 쭉 나열하는데 엄청나게 많다.   
그것들중 시간 많이걸리는 것들은 대략3가지  
- 인터럽트 처리
- 페이지 읽기
- 프로세스 재시작

다른것들은 어떻게 뭐 최적화하고 난리처러 줄이지만 페이지 읽기는 답이 없다  
혼자 단위가 다르다 다른것들 마이크로초,나노초 이럴때 혼자 밀리초다 대충 하드 한번 읽는데 3~5 밀리초가 걸린다고 한다.
즉 페이지 폴트 한번 나면 박살나게 효율이 떨어질것이다.  

그래서 페이지 폴트율을 낮게 유지하는것이 상당히 중요하다.

### 스왑 공간이 따로 있는 이유 — 그리고 왜 스왑이 빠른가

요구 페이징을 보면 자연스레 의문이 든다. **이미 디스크에 실행 파일이 있는데, 왜 굳이 "스왑 공간"이라는 걸 또 따로 두는 걸까?** 그냥 파일 시스템에 있는 파일을 그대로 읽어 쓰면 안 되나?

핵심은 **속도**다. 스왑 공간은 파일 시스템보다 빠르게 동작하도록 설계된 영역이다. 이유는 크게 두 가지.

**1. 파일 시스템 메타데이터 오버헤드가 없다.**
파일 시스템에서 데이터를 한 블록 읽으려면 디렉토리 탐색, inode 조회, 권한 확인, 블록 매핑 등 거쳐야 할 단계가 많다. 반면 스왑 공간은 **디스크 raw 영역에 직접 접근**한다. "어떤 페이지가 어느 디스크 블록에 있는지"만 알면 끝이고, 위의 파일 시스템 절차를 전부 건너뛴다.

**2. 페이지 단위에 최적화된 큰 블록 I/O를 한다.**
스왑은 페이지(보통 4KB) 단위로만 읽고 쓴다. 그래서 페이지를 디스크에 통째로 연속해서 배치하기 좋고, 시크(seek) 횟수도 줄어든다. 작은 블록을 잘게잘게 읽는 일반 파일 I/O보다 페이지 폴트 처리에 훨씬 유리한 구조다.

다만 빠른 만큼 귀해서, **스왑 공간은 보통 파일 시스템보다 작게 할당된다.** 그래서 OS는 스왑을 최대한 아껴 쓰는 전략을 쓴다. 이게 다음 절의 핵심이다.

### Linux / Windows 방식 — "스왑은 교체될 때만"

대부분의 데스크탑 OS는 다음 규칙을 따른다.

- **처음 페이지를 가져올 때**: 실행 파일이 있는 **파일 시스템에서 직접** 읽어온다. (스왑 안 씀)
- **메모리에서 쫓겨날 때만**: 그제서야 스왑 공간으로 보낸다.
- 쫓겨났던 페이지가 다시 필요해지면 → 스왑에서 읽어옴.

여기서 한 단계 더 영리한 최적화가 들어간다. **수정되지 않은 페이지(clean page)** 는 굳이 스왑에 쓸 필요가 없다. 원본이 파일 시스템에 그대로 남아있기 때문이다. 그냥 메모리에서 버리고, 나중에 필요해지면 파일에서 다시 읽으면 된다. 스왑은 오직 **수정된 페이지(dirty page)** 에만 써주면 된다.

| 페이지 종류 | 메모리에서 쫓겨날 때 | 다시 필요할 때 |
|---|---|---|
| 수정 안 됨 (clean / read-only) | 그냥 버림 | 파일 시스템에서 재독 |
| 수정됨 (dirty) | 스왑에 씀 | 스왑에서 읽음 |

이렇게 하면 귀한 스왑 공간을 진짜 필요할 때만 쓸 수 있다.

### BSD UNIX 방식 — 익명 메모리(anonymous memory)

또 다른 방식은 BSD UNIX 계열에서 쓰였다. 처음부터 파일 이미지를 메모리에 올리지 않고, **스왑 공간에 미리 충분한 공간을 잡아둔 뒤** 거기서 페이지를 가져오는 방식이다.

이때 등장하는 개념이 **익명 메모리(anonymous memory)** 다. 이름이 "익명"인 이유는 **파일 시스템의 특정 파일과 연결되어 있지 않기 때문**이다. 원본 파일이 없어서 출처가 익명. 그래서 anonymous.

> **안드로이드 관점.** `malloc()`으로 잡은 힙, 함수 스택, JVM 위에서 만든 모든 코틀린 객체들 — 죄다 익명 메모리다. 디스크에 원본 파일이 따로 없으니까. 반대로 APK 안의 `.dex` 코드 페이지는 파일 백업이 있는(file-backed) 메모리. 이 둘은 메모리 부족 상황에서 OS가 다르게 다룬다 (아래 모바일 절 참고).

### 모바일 OS — 스왑을 아예 안 쓴다

여기서 흐름이 갈린다. **iOS와 안드로이드 같은 모바일 OS는 스왑 공간을 사용하지 않는다.** (책에서는 9.5.3절에서 짧게 언급하고 10.7절에서 자세히 다룬다.)

왜 안 쓸까? 이유는 세 가지.

1. **플래시 저장소(NAND)는 쓰기 횟수에 수명 제한이 있다.** 스왑은 페이지를 빈번하게 쓰고 지우는 작업이라, 그대로 적용하면 저장소 수명이 빠르게 닳는다.
2. **모바일 저장 공간 자체가 작다.** 스왑 공간을 따로 크게 할당할 여유가 없다.
3. **배터리 소모.** 스왑 I/O가 잦으면 전력도 같이 닳는다.

그럼 메모리가 부족할 땐 어떻게 할까? 모바일 OS의 전략은 단순하다.

- **파일 백업이 있는 페이지(코드, 리소스 등 read-only)** → 그냥 버린다. 필요해지면 APK/IPA에서 다시 읽으면 됨.
- **수정된 페이지(dirty page)** → 버릴 수 없으니, 앱한테 직접 부탁한다.
  - 안드로이드: `onTrimMemory()`, `onLowMemory()`
  - iOS: `didReceiveMemoryWarning`
- **그래도 부족하면** → **앱을 강제 종료**한다. 안드로이드의 **LMK(Low Memory Killer)** 가 우선순위 낮은 백그라운드 앱부터 죽인다.

데스크탑이라면 "스왑으로 잠깐 옮겨두자"로 끝났을 상황이, 모바일에서는 **"앱 죽이기"** 까지 이어진다. 그래서 안드로이드 앱에서 `onTrimMemory()` 콜백 처리는 무시하면 안 된다. **OS가 "메모리 좀 비워줘"라고 정중하게 부탁하는 마지막 단계**이기 때문이다. 이걸 무시하면 다음 단계는 강제 종료다.

# 모바일 OS에서 스왑 공간을 사용하지 않는다는 말은 지금도 맞을까?

## Q1. 파일시스템이란 무엇인가요?

파일시스템은 저장장치에 데이터를 파일과 폴더 형태로 저장하고 관리하는 운영체제의 구조입니다.

디스크나 플래시 저장소는 원래 단순한 저장 공간일 뿐입니다. 운영체제는 파일시스템을 통해 파일 이름, 위치, 크기, 생성 시간, 수정 시간, 권한, 폴더 구조 등을 관리합니다.

예를 들어 사용자는 report.pdf라는 하나의 파일로 보지만, 실제 저장장치에는 그 파일의 데이터가 여러 블록에 나뉘어 저장될 수 있습니다. 파일시스템은 “이 파일이 저장장치의 어느 위치들에 저장되어 있는지”를 기억하고 관리합니다.

대표적인 파일시스템에는 Windows의 NTFS, macOS의 APFS, Linux의 ext4, Android에서 사용되는 ext4나 F2FS 등이 있습니다.

정리하면, 파일시스템은 저장장치 위에 파일과 폴더라는 질서를 만들어주는 운영체제의 저장 관리 방식입니다.

---

## Q2. 스왑 공간이란 무엇인가요?

스왑 공간은 RAM이 부족할 때, 당장 자주 사용하지 않는 메모리 내용을 임시로 옮겨두기 위한 공간입니다.

프로그램이 실행되면 RAM을 사용합니다. 그런데 RAM은 용량이 제한되어 있기 때문에, 여러 앱이나 프로그램이 동시에 많은 메모리를 사용하면 부족해질 수 있습니다.

이때 운영체제는 덜 사용되는 메모리 페이지를 다른 공간으로 잠시 옮겨 RAM을 확보합니다. 이때 사용하는 공간을 스왑 공간이라고 합니다.

예를 들어 RAM에 다음과 같은 내용이 있다고 가정할 수 있습니다.

text RAM ├─ 현재 사용 중인 앱 A ├─ 현재 사용 중인 앱 B └─ 오래 사용하지 않은 앱 C의 일부 메모리 

운영체제는 앱 C의 일부 메모리를 스왑 공간으로 옮겨 RAM을 비울 수 있습니다. 이 과정을 스왑 아웃이라고 하고, 나중에 다시 필요해져서 RAM으로 가져오는 과정을 스왑 인이라고 합니다.

정리하면, 스왑 공간은 RAM이 부족할 때 메모리 내용을 잠시 대피시키는 임시 공간입니다.

---

## Q3. 파일시스템과 스왑 공간은 어떻게 다른가요?

파일시스템과 스왑 공간은 둘 다 저장장치와 관련될 수 있지만, 목적이 다릅니다.

파일시스템은 사용자의 파일, 앱 데이터, 운영체제 파일 등을 저장하고 관리하기 위한 구조입니다. 반면 스왑 공간은 실행 중인 프로세스의 메모리 일부를 임시로 옮겨두기 위한 공간입니다.

즉 파일시스템은 파일 저장을 위한 체계이고, 스왑 공간은 메모리 부족을 보완하기 위한 공간입니다.

비유하면 다음과 같습니다.

text RAM = 책상 저장장치 = 책장 파일시스템 = 책장에 책을 분류하고 정리하는 방식 스왑 공간 = 책상이 좁을 때 잠시 치워두는 임시 보관함 

---

## Q4. 모바일 OS는 스왑 공간을 아예 사용하지 않나요?

전통적인 의미에서는 “모바일 OS는 스왑을 적극적으로 사용하지 않는다”는 설명이 어느 정도 맞습니다.

하지만 최신 환경까지 고려하면, “모바일 OS는 스왑을 전혀 사용하지 않는다”고 말하는 것은 정확하지 않습니다.

여기서 구분이 필요합니다.

text 전통적인 PC식 스왑: RAM 부족 → 메모리 페이지를 HDD나 SSD 같은 영구 저장소에 씀  최신 모바일에서 흔한 방식: RAM 부족 → 메모리 페이지를 RAM 안에서 압축해서 보관함 

안드로이드에서는 zRAM이라는 방식을 사용합니다. zRAM은 디스크나 플래시 저장소에 메모리 페이지를 쓰는 것이 아니라, RAM의 일부를 압축된 스왑 영역처럼 사용하는 방식입니다.

즉 안드로이드는 스왑을 완전히 안 쓴다기보다는, 영구 저장소에 쓰는 전통적인 스왑은 피하고, RAM 압축 기반의 스왑을 사용한다고 이해하는 편이 더 정확합니다.

---

## Q5. zRAM은 무엇인가요?

zRAM은 RAM 안에 압축된 메모리 영역을 만들어 사용하는 기술입니다.

일반적으로 스왑이라고 하면 저장장치에 메모리 페이지를 쓰는 방식을 떠올리지만, zRAM은 저장장치가 아니라 RAM 자체를 사용합니다. 대신 데이터를 압축해서 보관하기 때문에 같은 RAM 공간을 더 효율적으로 사용할 수 있습니다.

예를 들어 어떤 앱의 메모리 페이지 100MB가 당장 자주 사용되지 않는다고 가정해보겠습니다. 이 데이터를 zRAM에 압축해서 40MB 정도로 보관할 수 있다면, 결과적으로 RAM을 60MB 정도 아낀 것처럼 동작할 수 있습니다.

text 압축 전: RAM 100MB 사용 압축 후: zRAM 40MB 사용 절약 효과: 약 60MB 

물론 압축과 해제에는 CPU 연산이 필요합니다. 따라서 zRAM은 RAM 사용량을 줄이는 장점이 있지만, CPU 사용량과 배터리 소모라는 비용도 함께 가질 수 있습니다.

---

## Q6. 스마트폰에는 HDD가 없나요?

현대 스마트폰에는 일반적으로 HDD가 없습니다.

과거에는 아주 작은 하드디스크가 일부 모바일 기기에 사용된 사례가 있었지만, 현재 스마트폰의 내부 저장소는 대부분 NAND 플래시 기반입니다.

PC에서 사용하는 SSD도 NAND 플래시 기반이고, 스마트폰의 내부 저장소도 NAND 플래시 기반입니다. 다만 PC에서는 SATA SSD나 NVMe SSD 같은 형태가 흔하고, 스마트폰에서는 eMMC나 UFS 같은 형태가 사용됩니다.

정리하면 다음과 같습니다.

text PC - HDD: 회전하는 자기 디스크 기반 저장장치 - SSD: NAND 플래시 기반 저장장치  스마트폰 - eMMC 또는 UFS - 내부적으로 NAND 플래시 기반 저장장치 

따라서 스마트폰의 영구 저장소도 SSD처럼 플래시 메모리 기반이라고 이해할 수 있습니다.

---

## Q7. 플래시 저장소라면 SSD만 달린 PC도 스왑 때문에 같은 문제가 생기지 않나요?

원리적으로는 맞습니다.

SSD도 NAND 플래시 기반이기 때문에 쓰기 횟수에 따른 수명 문제가 있습니다. 스왑이 자주 발생하면 SSD에 쓰기 작업이 늘어나고, 장기적으로는 저장장치 수명에 영향을 줄 수 있습니다.

하지만 PC에서는 이 문제가 모바일보다 덜 치명적으로 취급됩니다.

이유는 다음과 같습니다.

text 1. PC SSD는 보통 용량이 크다. 2. SSD 컨트롤러가 wear leveling으로 쓰기 위치를 분산한다. 3. PC는 모바일보다 전력과 발열 여유가 크다. 4. 데스크톱 PC는 배터리 제약이 거의 없다. 5. PC는 프로그램을 죽이는 것보다 느려져도 유지하는 쪽을 선호한다. 

PC는 메모리가 부족하더라도 실행 중인 프로그램이 갑자기 종료되는 것보다는, 조금 느려지더라도 유지되는 쪽이 더 자연스럽습니다.

반면 모바일은 화면 반응성, 배터리, 발열이 훨씬 중요합니다. 그래서 저장소 스왑으로 버티기보다는, 메모리 압박이 심하면 백그라운드 앱을 종료하는 방식을 더 적극적으로 사용합니다.

## Q8. 모바일 저장공간이 작아서 스왑 공간을 두기 어렵다는 말은 무슨 뜻인가요?

## Q9. 요즘 스마트폰은 저장공간이 PC와 비슷한데, 이 설명은 여전히 맞나요?

하지만 저장공간이 커졌다고 해서 PC처럼 영구 저장소 기반 스왑을 적극적으로 사용하는 것이 곧바로 좋은 선택이 되는 것은 아닙니다.

text 쓰기 수명 배터리 소모 발열 I/O 지연 화면 반응성 앱 생명주기 정책

그래서 요즘 모바일 환경에서는 저장공간 용량만이 핵심 문제가 아닙니다. 저장소가 커졌더라도, 배터리와 발열, 반응성 문제 때문에 PC식 스왑을 적극적으로 쓰기는 어렵습니다.

## Q10. 모바일 OS는 메모리가 부족하면 어떻게 처리하나요?

모바일 OS는 보통 다음과 같은 순서로 메모리 압박을 처리합니다.

text 1. 불필요한 캐시를 회수한다. 2. 덜 중요한 메모리 페이지를 압축한다. 3. zRAM 같은 압축 스왑 영역을 활용한다. 4. 그래도 부족하면 백그라운드 프로세스를 종료한다. 

안드로이드에서는 Low Memory Killer Daemon, 즉 lmkd가 중요한 역할을 합니다. 메모리 압박이 심해지면 현재 사용자에게 덜 중요한 백그라운드 프로세스를 종료해서 시스템이 계속 반응할 수 있게 합니다.

그래서 안드로이드에서 어떤 앱을 백그라운드에 두었다가 다시 열었을 때 앱이 처음부터 다시 시작되는 경우가 있습니다. 이는 메모리 부족 상황에서 운영체제가 해당 앱의 프로세스를 종료했기 때문일 수 있습니다.

## Q11. PC와 모바일은 왜 메모리 부족 상황을 다르게 처리하나요?

PC는 보통 다음을 더 중요하게 여깁니다.

text 실행 중인 프로그램을 최대한 유지한다. 조금 느려져도 프로그램이 갑자기 종료되지 않게 한다. 

반면 모바일은 다음을 더 중요하게 여깁니다.

text 화면 반응성을 유지한다. 배터리를 아낀다. 발열을 줄인다. 현재 사용 중인 앱을 우선한다. 백그라운드 앱은 필요하면 종료한다. 

그래서 PC는 스왑을 사용해서라도 프로그램을 유지하려는 경향이 강하고, 모바일은 백그라운드 앱을 종료해서라도 현재 화면의 반응성과 배터리 효율을 지키려는 경향이 강합니다.

## Q12. “모바일 OS는 스왑을 사용하지 않는다”는 문장은 어떻게 이해해야 하나요?

## 최종 정리

파일시스템은 저장장치에 파일과 폴더를 저장하고 관리하는 구조입니다. 스왑 공간은 RAM이 부족할 때 메모리 내용을 임시로 옮겨두기 위한 공간입니다.

스마트폰의 내부 저장소는 일반적으로 HDD가 아니라 NAND 플래시 기반입니다. PC의 SSD와 같은 계열의 저장 기술이라고 볼 수 있습니다.

따라서 스마트폰에서도 저장소 기반 스왑을 사용하면 플래시 쓰기 수명, 배터리 소모, 발열, I/O 지연 문제가 생길 수 있습니다. 이 문제는 SSD만 사용하는 PC에서도 원리적으로는 존재하지만, PC는 전력과 발열, 용량 측면에서 상대적으로 여유가 있고 프로그램 유지가 더 중요하기 때문에 스왑을 더 적극적으로 사용합니다.

반면 모바일 OS는 화면 반응성, 배터리, 발열을 더 중요하게 보기 때문에 전통적인 저장소 기반 스왑을 적극적으로 사용하지 않습니다. 대신 zRAM 같은 압축 메모리 기반 스왑과 백그라운드 프로세스 종료를 조합해 메모리 부족 상황을 처리합니다.

## 쓰기 시 복사 (Copy-on-Write, COW)

`fork()`로 자식을 만들 때 **부모의 페이지를 곧바로 복사하지 않고 일단 같이 쓴다.** 그러다 둘 중 한쪽이 그 페이지에 **쓰기를 시도하는 순간에만** 그 페이지의 복사본을 만든다. 이게 쓰기 시 복사다.

왜 이렇게 할까? 옛날 방식은 `fork()` 때 부모 페이지를 전부 복사해서 자식 주소 공간을 만들었는데, **자식은 보통 만들어지자마자 `exec()`로 새 프로그램으로 갈아치운다.** 그러면 애써 복사한 페이지가 전부 쓸모없어진다. 그러니 복사를 미루는 게 이득이다.

**동작 방식**
- 공유 중인 페이지는 "쓰기 시 복사 페이지"로 표시(사실상 read-only)된다.
- 한쪽이 쓰기를 시도 → 가용 프레임 리스트에서 빈 프레임을 얻어 **그 페이지만** 복사 → 쓴 프로세스의 주소 공간에 사상.
- 결국 **수정한 페이지만** 복사본이 생기고, 안 건드린 페이지(특히 코드 같은 read-only 페이지)는 부모·자식이 계속 공유한다.

> 그림 10.7 → 10.8: 프로세스 1이 페이지 C를 수정하기 *전*에는 A·B·C를 부모와 공유하다가, C에 쓰는 순간 "페이지 C 복사"가 생겨 프로세스 1만 그걸 가리킨다.

### vfork() — COW를 안 쓰는 위험한 버전

`vfork()`(virtual memory fork)는 `fork()`와 다르게 동작한다.
- **부모를 잠시 멈추고(suspend), 자식이 부모의 주소 공간을 그대로 빌려 쓴다.** 페이지를 아예 복사하지 않는다.
- 따라서 **자식이 부모 주소 공간의 페이지를 수정하면, 부모가 재개됐을 때 그 변경이 그대로 보인다.** 그래서 자식은 부모 메모리를 건드리면 안 된다.
- 자식이 만들어지자마자 `exec()`하는 경우 전용. 복사가 0이라 매우 빠르고, UNIX 셸(명령어 해석기) 구현에 쓰인다.

<details>
<summary>📖 책에 없는 내용 — 안드로이드의 Zygote와 COW</summary>

안드로이드에서 **모든 앱 프로세스는 Zygote라는 부모 프로세스를 `fork()`해서 생긴다.** Zygote는 부팅 때 미리 ART 런타임과 자주 쓰는 프레임워크 클래스·리소스를 메모리에 올려둔 "원형(原型)" 프로세스다.

- 앱을 실행하면 Zygote를 fork → COW 덕분에 **프레임워크 페이지를 복사하지 않고 공유**한다. 그래서 앱 실행이 빠르고, 수십 개 앱이 떠 있어도 공통 부분은 메모리에 한 벌만 존재한다.
- 앱이 그 공유 페이지에 쓰기를 하면 그때만 복사본이 생긴다(= COW).
- 이래서 안드로이드 메모리를 볼 때 **PSS(공유분을 나눠서 계산)** 와 **USS(이 프로세스만의 메모리)** 를 구분한다. `dumpsys meminfo`에서 보이는 그 값들이 COW로 공유되는 메모리 때문에 나뉘는 것.

요약: 책의 "fork는 COW로 빨라진다"가 안드로이드에서는 **Zygote 구조 = 앱 시작 속도 + 메모리 절약**의 토대다.

## 페이지 교체 (Page Replacement)

지금까지는 "각 페이지는 처음 접근될 때 한 번만 폴트가 난다"고 가정했는데, 현실은 그렇게 깔끔하지 않다. 문제의 출발점은 **메모리 과할당(over-allocation)** 이다.

**과할당이 왜 생기나**
- 프로세스는 보통 자기 페이지의 일부만 실제로 쓴다. 그래서 OS는 물리 프레임보다 더 많은 페이지를 "필요할 때 올리면 되지" 하고 다중 프로그래밍 정도를 높인다. (예: 40프레임으로 8개 프로세스를 돌림)
- 그런데 운 나쁘게 **여러 프로세스가 동시에 자기 페이지를 거의 다 요구**하면, 프레임이 모자란다. 게다가 I/O 버퍼도 상당한 메모리를 먹는다.
- 즉, 페이지 폴트가 났는데 **빈 프레임이 하나도 없는** 상황이 발생한다(그림 10.9의 물음표).

**프레임이 없을 때 OS의 선택지**
1. 프로세스 종료 → 페이징은 사용자에게 투명해야 하므로 나쁜 선택.
2. 표준 스와핑(프로세스 통째로 swap-out) → 오버헤드가 커서 요즘 OS는 거의 안 씀.
3. **페이지 교체(page replacement)** → 현실적인 답. 아래부터 이걸 다룬다.

### 기본적 페이지 교체 (10.4.1)

빈 프레임이 없으면, **현재 메모리에 있는 페이지 중 하나(희생자, victim)를 골라 쫓아내고** 그 자리에 새 페이지를 넣는다.

**페이지 폴트 + 교체 처리 순서**
1. 보조저장장치에서 필요한 페이지 위치를 찾는다.
2. 빈 프레임을 찾는다.
   - 있으면 → 그냥 사용.
   - 없으면 → **페이지 교체 알고리즘**으로 victim 프레임을 고른다.
   - victim 페이지를 (필요하면) 보조저장장치에 기록하고 테이블 갱신.
3. 빼앗은 프레임에 새 페이지를 읽어오고 테이블 갱신.
4. 폴트가 난 지점부터 프로세스 재개.

**핵심 비용 문제:** 빈 프레임이 없으면 디스크를 **두 번** 친다 — victim 내보내기(write) + 새 페이지 읽기(read). 폴트 서비스 시간이 2배가 된다.

**변경 비트(modify bit / dirty bit)로 절반 절약**
- CPU가 그 페이지에 한 바이트라도 쓰면 dirty bit가 켜진다.
- victim을 쫓아낼 때 dirty bit를 확인:
  - **clean(안 바뀜)** → 보조저장장치에 원본이 그대로 있으니 **그냥 버린다.** write 생략 → I/O 절반.
  - **dirty(바뀜)** → 보조저장장치에 기록해야 함.
- 코드처럼 read-only인 페이지는 항상 clean이라 언제든 그냥 버릴 수 있다.

> 이게 앞 절의 "clean page는 스왑에 안 쓰고 버린다"와 같은 얘기다. 모바일에서 read-only 코드/리소스 페이지를 그냥 버리고 APK에서 다시 읽는 것도 정확히 이 원리.

### FIFO 페이지 교체 (10.4.2)

**먼저 참조열(reference string) 개념부터.** 폴트 횟수를 계산하려면 실제 주소 전체가 아니라 **페이지 번호의 나열**만 보면 된다. 같은 페이지를 연속 참조하면 두 번째부터는 폴트가 안 나므로 무시. (예: 주소열을 페이지 크기로 나눠 `1,4,1,6,1,6,...`처럼 축약)

일반적으로 **프레임 수가 늘면 폴트 수는 줄어든다**(그림 10.11의 우하향 곡선).

**FIFO 알고리즘**
- 가장 간단한 교체 알고리즘. **메모리에 가장 먼저 올라온(오래된) 페이지를 쫓아낸다.**
- 구현: 페이지들을 올라온 순서대로 **큐(FIFO queue)** 에 넣고, 교체가 필요하면 큐의 머리를 빼고 새 페이지는 꼬리에 넣는다.
- 예시 참조열 `7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1` + 3프레임 → **폴트 15회**(그림 10.12).

**FIFO의 단점**
- "오래된 페이지 = 이제 안 쓰는 페이지"가 아니다. **자주 쓰는 변수가 담긴 페이지도 단지 오래됐다는 이유로 쫓겨날 수 있다.** 그러면 곧바로 다시 폴트가 나서 도로 불러온다 → 폴트율 상승, 속도 저하.

**Belady의 모순 (Belady's anomaly)**
- 상식적으로 **프레임을 더 주면 폴트가 줄어야** 한다. 그런데 FIFO에서는 **프레임을 늘렸는데 폴트가 오히려 늘어나는** 경우가 있다.
- 참조열 `1,2,3,4,1,2,5,1,2,3,4,5`에서: **3프레임 → 9폴트, 4프레임 → 10폴트**(그림 10.13). 프레임을 늘렸는데 더 나빠진다.
- 이 황당한 현상이 Belady의 모순이고, "더 똑똑한 교체 정책이 필요하다"는 동기가 된다 → 다음 절 **최적 페이지 교체(OPT)**.

<details>
<summary>📖 책에 없는 내용 — 실제 리눅스/안드로이드는 FIFO를 안 쓴다</summary>

책은 설명용으로 FIFO를 먼저 보여주지만, **실제 리눅스 커널(=안드로이드)은 FIFO를 쓰지 않는다.** Belady의 모순이 있고 성능이 나빠서다.

- 리눅스는 **LRU를 근사한(approximate) 알고리즘**을 쓴다. 페이지를 **active list / inactive list** 두 개로 나눠 관리하고, 최근에 접근된(accessed bit가 켜진) 페이지는 active 쪽으로 승격, 오래 안 쓰인 페이지는 inactive로 내려보내 회수 후보로 삼는다. (책 뒷부분 LRU 절과 연결됨)
- 안드로이드는 여기에 더해 앞 절에서 본 대로 **스왑이 없으니** → clean한 file-backed 페이지(앱 코드·리소스)는 회수해서 버리고, dirty한 anonymous 페이지는 버릴 수 없어 `onTrimMemory()` → 그래도 모자라면 **LMK가 앱을 종료**한다.

정리: "교체 알고리즘"이라는 개념은 동일하지만, victim을 고르는 기준이 FIFO가 아니라 **'최근에 안 쓰인 것(LRU 근사)'** 이고, 모바일에선 교체의 종착지가 swap이 아니라 **앱 kill**이라는 점이 데스크탑과 다르다.

</details>

### 최적 페이지 교체 (Optimal, OPT / MIN) (10.4.3)

Belady의 모순을 보고 나온 질문: "그럼 **모순도 없고 폴트율도 최저인** 완벽한 알고리즘이 있을까?" → 있다. 그게 **최적(optimal) 페이지 교체**, 줄여서 OPT 또는 MIN이다.

**규칙은 한 줄이다.**
> **앞으로 가장 오랫동안 사용되지 않을 페이지를 victim으로 골라라.** (미래에 가장 늦게 다시 참조될 페이지를 내쫓는다.)

- 같은 참조열 `7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1` + 3프레임 → **폴트 9회**(그림 10.14). FIFO의 15회보다 훨씬 적고, **어떤 알고리즘도 같은 조건에서 9회보다 적게 낼 수 없다.** 그래서 "최적".
  - 예: 페이지 7을 가장 먼저 내쫓는다 — 7은 18번째 참조에서야 다시 쓰이는데, 0(5번째)·1(14번째)보다 훨씬 나중이기 때문.

**치명적 한계 — 구현 불가능**
- victim을 고르려면 **프로세스가 앞으로 메모리를 어떻게 참조할지 미리 알아야** 한다. 미래를 알 수 없으니 실제로는 못 만든다.
- (5장에서 본 **SJF CPU 스케줄링**이 "다음 CPU burst 길이를 미리 알아야 해서" 구현 불가능했던 것과 똑같은 이유다.)

**그럼 왜 배우나 — 벤치마크 기준선**
- 실전 알고리즘이 아니라 **"이론상 최선"의 잣대**로 쓴다. 새 알고리즘을 만들면 "OPT 대비 몇 % 나쁜가"로 평가한다. (예: 최악의 경우에도 OPT보다 12.3% 이상 나빠지지 않고 평균 4.7% 차이뿐이라면 충분히 좋은 알고리즘.)
- 그래서 다음 절부터는 **"과거 기록으로 미래를 근사하는"** 현실적인 알고리즘(LRU 등)으로 넘어간다.

<details>
<summary>📖 책에 없는 내용 — OPT가 알려주는 한 가지 관점</summary>

OPT의 진짜 교훈은 **"모든 현실 교체 알고리즘은 결국 미래를 과거로 추측하는 게임"** 이라는 것이다.

- OPT = 미래를 안다(불가능). → LRU = "최근에 많이 썼으면 앞으로도 쓰겠지"라고 **과거로 미래를 추측**. → 리눅스/안드로이드의 active·inactive list나 second-chance(clock)는 그 LRU를 더 싸게 근사한 것.
- 안드로이드 앱 입장에서도 같은 원리가 보인다: **시스템이 어떤 앱/페이지를 살릴지 죽일지도 "과거 사용 패턴"으로 추측**한다. 최근에 포그라운드였던 앱, 자주 쓰이는 페이지는 LMK에서 늦게 죽는다. 그래서 `onTrimMemory()`에서 캐시를 잘 비워 USS를 낮춰두면, 시스템의 "추측"에서 살아남을 확률이 올라간다.

즉 OPT는 "절대 못 만드는 알고리즘"이지만, **왜 캐시·메모리 관리가 늘 '사용 빈도/최근성' 기반인지**를 설명해주는 출발점이다.

</details>
