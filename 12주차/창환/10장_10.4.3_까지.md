# 가상 메모리

## 가상 메모리란?

**가상 메모리(Virtual Memory)** 는 프로세스가 실제 물리 메모리(RAM)의 크기나 위치에 얽매이지 않고, 자신만의 연속적이고 거대한 메모리 공간(=논리 주소 공간)을 사용하는 것처럼 보이게 해주는 기법이다.

쉽게 말하면, **"프로세스가 보는 메모리"와 "실제 RAM"을 분리**해서, 그 사이를 운영체제와 하드웨어(MMU)가 알아서 매핑해 주는 구조다. 프로세스는 자기가 RAM 어디에 올라가 있는지, 심지어 지금 RAM에 올라가 있긴 한지조차 신경 쓸 필요가 없다.

## 가상 메모리를 사용해서 얻는 이점

### 1. 논리 메모리와 물리 메모리의 분리
프로세스는 자신만의 깔끔한 주소 공간(예: 0번지부터 시작하는 연속된 공간)을 본다. 실제로는 물리 메모리 여기저기 흩어져 있어도, 프로그래머와 프로세스 입장에서는 신경 쓸 필요가 없다. 덕분에 **프로그래밍이 단순해지고**, OS는 물리 메모리를 유연하게 관리할 수 있다.

### 2. 물리 메모리보다 큰 프로세스 실행 가능
**요구 페이징(demand paging)** 덕분에, 프로세스 전체를 한 번에 메모리에 올릴 필요가 없다. 당장 실행에 필요한 페이지만 RAM에 올리고, 나머지는 디스크(스왑 공간)에 두었다가 필요할 때 가져온다.

> ※ 동적 적재(dynamic loading)와 혼동하기 쉽다. 동적 적재는 **프로그래머가 직접** 루틴 단위로 적재 시점을 관리하는 기법이고, 가상 메모리의 요구 페이징은 **OS가 자동으로** 페이지 단위로 처리해 준다.
>
> 예를 들어 C 언어에서는 `dlopen()`, `dlsym()`(Unix/Linux)이나 `LoadLibrary()`(Windows) 같은 API를 **프로그래머가 직접 호출**해서 "이 루틴을 지금 메모리에 올려라"라고 명시해 줘야 했다. 더 옛날에는 **오버레이(overlay)** 구조를 직접 설계해서 같은 메모리 영역에 서로 다른 모듈을 번갈아 올리기도 했다. 즉 동적 적재는 어디까지나 **사용자 수준의 수동 기법**이다.
>
> 반면 가상 메모리는 프로그래머가 신경 쓰지 않아도 OS가 알아서 처리해 주므로, "물리 메모리보다 큰 프로그램 실행"은 본질적으로 **가상 메모리가 제공하는 이점**이라고 보는 게 맞다.

### 3. 페이지 공유를 통한 프로세스 간 메모리 공유
서로 다른 프로세스가 동일한 물리 페이지를 자신의 논리 주소 공간에 매핑할 수 있다. 이를 통해
- 공유 라이브러리(예: libc)를 여러 프로세스가 메모리에 한 벌만 올려놓고 같이 쓰고,
- 프로세스 간 통신(IPC)에 공유 메모리를 활용할 수 있다.

메모리 절약과 효율적인 협업이 동시에 가능해진다.

### 4. 프로세스 생성(fork) 속도 향상
`fork()`로 자식 프로세스를 만들 때, 부모의 메모리 전체를 복사하지 않고 **페이지 테이블만 복사**한 뒤 같은 물리 페이지를 가리키게 한다. 이후 어느 한쪽이 그 페이지에 쓰기를 시도하는 순간에만 페이지를 복제한다(**Copy-on-Write, COW**).

대부분의 페이지는 읽기만 하거나, 자식이 곧바로 `exec()`을 호출해 새 프로그램으로 갈아치우기 때문에, **실제로 복사되는 페이지는 극히 일부**다. 결과적으로 프로세스 생성 비용이 크게 줄어든다.


### 잠깐, 페이징이랑 가상 메모리는 결국 같은 거 아닌가? (AI Q&A)

위에서 나열한 이점 중 일부가 사실은 이전 장에서 다룬 **페이징(paging)** 자체의 이점처럼 보일 수 있다. 헷갈리는 부분을 AI에게 물어본 내용을 정리했다.

**Q.** 페이징만으로도 프로세스를 쪼개서 메모리에 올리는 것도 되고, 논리 주소를 물리 주소로 매핑하는 것도 다 되잖아? 그럼 "논리 메모리와 물리 메모리의 분리"나 "물리 메모리보다 큰 프로세스 실행" 같은 건 사실 가상 메모리만의 이점이 아니라 페이징의 이점 아닌가? 어차피 MMU(메모리 관리 장치)가 알아서 매핑해 주는 거고, 가상 메모리랑은 직접적인 상관이 없는 것 같은데. "우리가 신경 안 써도 된다"는 점이 가상 메모리의 핵심인 건가?

**A.** 직관의 절반은 정확하다. 하나씩 짚어보면:

- **"논리 메모리와 물리 메모리의 분리"** → 이건 엄밀히 말하면 **페이징의 이점**이다. 페이지 테이블과 MMU만 있어도 이미 달성된다. 책에서 가상 메모리의 이점으로 같이 묶어 적는 경우가 많은데, 가상 메모리는 그 위에서 분리 효과를 더 극단적으로 활용하는 것에 가깝다.

- **"물리 메모리보다 큰 프로세스 실행"** → 이건 페이징만으로는 **불가능**하다. 순수 페이징은 여전히 *프로세스의 모든 페이지가 RAM에 올라가 있어야* 실행할 수 있기 때문이다. 100MB짜리 프로세스를 50MB RAM에서 돌리려면, 디스크(스왑)와 페이지 폴트 처리 메커니즘이 필요하고, 이게 바로 가상 메모리가 새로 도입하는 영역이다.

즉, **페이징은 "RAM 안에서 흩어진 페이지를 모아 보여주는" 기법**이라면, **가상 메모리는 그 페이지 일부를 디스크에까지 확장한 기법**이라고 보면 깔끔하다.

"우리가 신경 안 써도 된다"는 표현도 사실 정확하다. 프로그래머/프로세스 입장에서 페이지 폴트도, 스왑도, 교체도 보이지 않게 OS가 처리해 주기 때문에, 결과적으로는 "그냥 메모리가 큰 것처럼 보인다"가 가상 메모리의 사용자 측 효과다.

> **비유.** 페이징은 "책장에 책을 흩어 놔도, 카탈로그(페이지 테이블)로 책 위치를 찾을 수 있다"는 것이고, 가상 메모리는 "책장 용량보다 책이 많아도, 자주 안 보는 건 창고(디스크)에 두었다가 필요할 때 꺼내온다"는 것. 창고와 꺼내오는 로직이 가상 메모리의 정체.

## 요구 페이징

**요구 페이징(demand paging)** 은 말 그대로 "**요구(접근)될 때** 페이지를 메모리에 올리는" 방식이다. 프로세스를 시작할 때 모든 페이지를 미리 메모리에 적재하지 않고, 실제로 그 페이지가 참조되는 순간에 비로소 디스크에서 가져온다.

장점은 명확하다.
- 프로세스 시작이 빠르다 (전부 적재할 필요 없음)
- 메모리를 아낄 수 있다 (안 쓰는 페이지는 안 올림)
- 더 많은 프로세스를 동시에 돌릴 수 있다

### 유효/무효 비트의 의미 확장

이 비트는 새로운 게 아니다. **9장 페이징에서 등장했던 그 비트를 그대로 재사용**하되, 의미만 한 단계 더 확장한다.

| | 9장 (페이징) | 10장 (가상 메모리 / 요구 페이징) |
|---|---|---|
| **유효(valid)** | 이 프로세스의 합법적 주소 공간에 속함 | 합법적 주소 공간에 속함 **+ 현재 RAM에 적재되어 있음** |
| **무효(invalid)** | 프로세스 주소 공간 밖 (불법 접근) | (a) 주소 공간 밖이거나, (b) 합법적이지만 **지금은 디스크에 있음** |

비트는 1개로 그대로지만, 무효의 의미가 (a) "쓰면 안 되는 페이지"와 (b) "써도 되는데 아직 메모리에 안 올라온 페이지" 둘로 늘어난 셈이다. 둘의 구분은 무효 트랩이 발생한 뒤 OS가 PCB의 합법 주소 범위 등을 참고해서 판단한다.

- (a)면 → **프로세스 중단(Segmentation Fault)**
- (b)면 → **페이지 폴트(page fault)** 처리

### 페이지 폴트 트랩 처리 순서

1. 무효로 표시된 페이지에 접근 → MMU가 트랩 발생
2. OS가 트랩을 받아 합법적 참조인지 확인
3. 빈 프레임을 찾는다 (없으면 페이지 교체 알고리즘으로 하나 비움)
4. 디스크(스왑 공간)에서 해당 페이지를 빈 프레임으로 읽어옴
5. 페이지 테이블 갱신, 유효 비트 set
6. **트랩을 일으킨 명령어를 처음부터 다시 실행(restart)**

프로세스 입장에서는 잠깐 멈췄다 다시 실행될 뿐, 폴트가 일어났는지조차 모른다.

### 순수 요구 페이징 (Pure Demand Paging)

극단적으로는 **처음에 단 한 페이지도 미리 안 올리는** 방식도 가능하다. 프로세스가 첫 명령어를 fetch하려는 순간부터 페이지 폴트가 터지고, 거기서부터 필요한 페이지가 하나씩 올라온다. 이론적으로 메모리를 가장 아끼지만, 시작 직후 폴트가 폭주해서 비효율적이라 실제로는 잘 쓰이지 않는다.

### 그런데 왜 폴트가 매번 안 터지나? — 참조의 지역성

"매번 페이지 폴트가 나면 디스크 I/O 때문에 엄청 느릴 텐데?" 라는 의문이 들 수 있다. 하지만 실제로는 페이지 폴트가 **드물게만** 발생한다. 그 이유가 **참조의 지역성(locality of reference)** 이다.

- **시간적 지역성**: 최근에 접근한 메모리는 곧 다시 접근될 가능성이 높다 (반복문, 자주 호출되는 함수)
- **공간적 지역성**: 어떤 주소에 접근하면 근처 주소도 곧 접근된다 (배열 순회, 순차적 코드 실행)

→ 한 번 메모리에 올라온 페이지가 한동안 계속 쓰이기 때문에, 폴트는 가끔만 일어난다. 그래서 요구 페이징이 **실용적으로 잘 작동**하는 것이다. (지역성이 깨지는 워크로드는 폴트가 폭주하는데, 이걸 **스래싱(thrashing)** 이라고 부르고 뒷장에서 따로 다룬다.)

### 명령어 재시작 문제 — 메모리를 많이 건드리는 명령어의 함정

페이지 폴트가 나면 OS는 트랩을 일으킨 명령어를 **처음부터 다시 실행**한다. 보통은 문제없지만, **한 명령어가 여러 메모리 위치를 건드리는 경우**가 까다롭다.

예를 들어 어떤 CPU의 **블록 이동 명령**(예: IBM의 MVC — 256바이트 메모리 블록을 src에서 dst로 복사):
- 100바이트 복사한 시점에 dst의 다음 페이지에서 폴트 발생
- 페이지 가져온 뒤 명령어를 "처음부터" 재실행
- 그런데 **src와 dst가 겹쳐 있다면**? 이미 덮어쓴 영역의 값이 달라져 있어서, 재실행하면 결과가 망가진다.

3-주소 명령어(예: `ADD R1, R2, R3` 형태로 여러 피연산자를 읽고 결과를 또 다른 위치에 쓰는 명령)도 비슷한 위험이 있다.

**해결책 (CPU/OS가 함께 처리):**
1. **선검사(pre-check)**: 명령어 시작 전, 양쪽 페이지를 미리 건드려서 폴트가 날 거면 *먼저* 일으켜 둠 → 본 실행 중에는 폴트 안 남
2. **임시 레지스터 백업**: 덮어쓰기 전 원본을 백업해두고, 폴트 발생 시 롤백
3. **재시작 가능 명령어(restartable instruction)** 설계: 명령어를 항상 처음부터 안전하게 재실행해도 결과가 같도록 하드웨어 차원에서 보장

**시사점:** 가상 메모리를 지원하는 CPU는 이런 메커니즘을 반드시 갖춰야 한다. 그래서 **메모리를 많이 건드리는 명령어(블록 이동, 문자열 처리 등)는 설계할 때 페이지 폴트 발생 가능성을 항상 고려**해야 한다.

### 가용 프레임 리스트 (free-frame list)

페이지 폴트가 발생하면 OS는 디스크에서 페이지를 읽어와 **어딘가 빈 프레임**에 넣어야 한다. 이때 "빈 프레임이 어디 있는지" 매번 메모리를 훑어 찾는 건 비효율이라, OS는 미리 **사용 가능한(비어 있는) 프레임 목록을 연결 리스트로 관리**해 둔다. 이게 가용 프레임 리스트다.

- 페이지 폴트 발생 → 리스트에서 한 프레임을 떼어서 사용
- 프로세스가 종료되거나 페이지가 교체되어 비워지면 → 다시 리스트에 반환
- 부팅 시점에는 거의 모든 프레임이 이 리스트에 들어있고, 시간이 지나면서 사용/반환을 반복

### Zero-Fill-on-Demand

가용 프레임 리스트에서 프레임 하나를 떼어서 프로세스에게 주기 전에, OS는 그 프레임의 내용을 **모두 0으로 초기화**한다. 이를 **zero-fill-on-demand**라고 부른다.

왜 굳이 0으로 채울까? → **보안 때문이다.**
- 빈 프레임이라고는 하지만, 그 안에는 **이전에 그 자리를 썼던 다른 프로세스의 데이터(비밀번호, 토큰, 키 등)가 그대로 남아있을** 수 있다.
- 만약 0으로 초기화하지 않고 그대로 새 프로세스에게 넘기면, 새 프로세스는 자기 메모리를 읽는 척하면서 **이전 프로세스의 잔여 데이터를 훔쳐볼 수 있다.** 이건 명백한 정보 누출 취약점이다.
- 그래서 OS는 프레임을 넘기기 전에 0으로 싹 밀어버려서, 다른 프로세스의 흔적이 남지 않도록 보장한다.

이름이 "on-demand"인 이유는, 미리 모든 빈 프레임을 0으로 채워두는 게 아니라 **실제로 프로세스에게 줄 때(=요구 시점에)** 그 프레임만 0으로 초기화하기 때문이다.

### 요구 페이징 성능
책에서 페이지 폴트 과정을 쭉 나열하는데 엄청나게 많다.   
그것들중 시간 많이걸리는 것들은 대략3가지  
- 인터럽트 처리
- 페이지 읽기
- 프로세스 재시작

다른것들은 어떻게 뭐 최적화하고 난리처러 줄이지만 페이지 읽기는 답이 없다  
혼자 단위가 다르다 다른것들 마이크로초,나노초 이럴때 혼자 밀리초다 대충 하드 한번 읽는데 3~5 밀리초가 걸린다고 한다.
즉 페이지 폴트 한번 나면 박살나게 효율이 떨어질것이다.  

그래서 페이지 폴트율을 낮게 유지하는것이 상당히 중요하다.

### 스왑 공간이 따로 있는 이유 — 그리고 왜 스왑이 빠른가

요구 페이징을 보면 자연스레 의문이 든다. **이미 디스크에 실행 파일이 있는데, 왜 굳이 "스왑 공간"이라는 걸 또 따로 두는 걸까?** 그냥 파일 시스템에 있는 파일을 그대로 읽어 쓰면 안 되나?

핵심은 **속도**다. 스왑 공간은 파일 시스템보다 빠르게 동작하도록 설계된 영역이다. 이유는 크게 두 가지.

**1. 파일 시스템 메타데이터 오버헤드가 없다.**
파일 시스템에서 데이터를 한 블록 읽으려면 디렉토리 탐색, inode 조회, 권한 확인, 블록 매핑 등 거쳐야 할 단계가 많다. 반면 스왑 공간은 **디스크 raw 영역에 직접 접근**한다. "어떤 페이지가 어느 디스크 블록에 있는지"만 알면 끝이고, 위의 파일 시스템 절차를 전부 건너뛴다.

**2. 페이지 단위에 최적화된 큰 블록 I/O를 한다.**
스왑은 페이지(보통 4KB) 단위로만 읽고 쓴다. 그래서 페이지를 디스크에 통째로 연속해서 배치하기 좋고, 시크(seek) 횟수도 줄어든다. 작은 블록을 잘게잘게 읽는 일반 파일 I/O보다 페이지 폴트 처리에 훨씬 유리한 구조다.

다만 빠른 만큼 귀해서, **스왑 공간은 보통 파일 시스템보다 작게 할당된다.** 그래서 OS는 스왑을 최대한 아껴 쓰는 전략을 쓴다. 이게 다음 절의 핵심이다.

### Linux / Windows 방식 — "스왑은 교체될 때만"

대부분의 데스크탑 OS는 다음 규칙을 따른다.

- **처음 페이지를 가져올 때**: 실행 파일이 있는 **파일 시스템에서 직접** 읽어온다. (스왑 안 씀)
- **메모리에서 쫓겨날 때만**: 그제서야 스왑 공간으로 보낸다.
- 쫓겨났던 페이지가 다시 필요해지면 → 스왑에서 읽어옴.

여기서 한 단계 더 영리한 최적화가 들어간다. **수정되지 않은 페이지(clean page)** 는 굳이 스왑에 쓸 필요가 없다. 원본이 파일 시스템에 그대로 남아있기 때문이다. 그냥 메모리에서 버리고, 나중에 필요해지면 파일에서 다시 읽으면 된다. 스왑은 오직 **수정된 페이지(dirty page)** 에만 써주면 된다.

| 페이지 종류 | 메모리에서 쫓겨날 때 | 다시 필요할 때 |
|---|---|---|
| 수정 안 됨 (clean / read-only) | 그냥 버림 | 파일 시스템에서 재독 |
| 수정됨 (dirty) | 스왑에 씀 | 스왑에서 읽음 |

이렇게 하면 귀한 스왑 공간을 진짜 필요할 때만 쓸 수 있다.

### BSD UNIX 방식 — 익명 메모리(anonymous memory)

또 다른 방식은 BSD UNIX 계열에서 쓰였다. 처음부터 파일 이미지를 메모리에 올리지 않고, **스왑 공간에 미리 충분한 공간을 잡아둔 뒤** 거기서 페이지를 가져오는 방식이다.

이때 등장하는 개념이 **익명 메모리(anonymous memory)** 다. 이름이 "익명"인 이유는 **파일 시스템의 특정 파일과 연결되어 있지 않기 때문**이다. 원본 파일이 없어서 출처가 익명. 그래서 anonymous.

> **안드로이드 관점.** `malloc()`으로 잡은 힙, 함수 스택, JVM 위에서 만든 모든 코틀린 객체들 — 죄다 익명 메모리다. 디스크에 원본 파일이 따로 없으니까. 반대로 APK 안의 `.dex` 코드 페이지는 파일 백업이 있는(file-backed) 메모리. 이 둘은 메모리 부족 상황에서 OS가 다르게 다룬다 (아래 모바일 절 참고).

### 모바일 OS — 스왑을 아예 안 쓴다

여기서 흐름이 갈린다. **iOS와 안드로이드 같은 모바일 OS는 스왑 공간을 사용하지 않는다.** (책에서는 9.5.3절에서 짧게 언급하고 10.7절에서 자세히 다룬다.)

왜 안 쓸까? 이유는 세 가지.

1. **플래시 저장소(NAND)는 쓰기 횟수에 수명 제한이 있다.** 스왑은 페이지를 빈번하게 쓰고 지우는 작업이라, 그대로 적용하면 저장소 수명이 빠르게 닳는다.
2. **모바일 저장 공간 자체가 작다.** 스왑 공간을 따로 크게 할당할 여유가 없다.
3. **배터리 소모.** 스왑 I/O가 잦으면 전력도 같이 닳는다.

그럼 메모리가 부족할 땐 어떻게 할까? 모바일 OS의 전략은 단순하다.

- **파일 백업이 있는 페이지(코드, 리소스 등 read-only)** → 그냥 버린다. 필요해지면 APK/IPA에서 다시 읽으면 됨.
- **수정된 페이지(dirty page)** → 버릴 수 없으니, 앱한테 직접 부탁한다.
  - 안드로이드: `onTrimMemory()`, `onLowMemory()`
  - iOS: `didReceiveMemoryWarning`
- **그래도 부족하면** → **앱을 강제 종료**한다. 안드로이드의 **LMK(Low Memory Killer)** 가 우선순위 낮은 백그라운드 앱부터 죽인다.

데스크탑이라면 "스왑으로 잠깐 옮겨두자"로 끝났을 상황이, 모바일에서는 **"앱 죽이기"** 까지 이어진다. 그래서 안드로이드 앱에서 `onTrimMemory()` 콜백 처리는 무시하면 안 된다. **OS가 "메모리 좀 비워줘"라고 정중하게 부탁하는 마지막 단계**이기 때문이다. 이걸 무시하면 다음 단계는 강제 종료다.

### 10.7절 — 압축 메모리 (참고)

스왑을 못 쓰는 모바일이 메모리를 그래도 효율적으로 쓰려고 선택한 대안이 **압축 메모리(compressed memory)** 다. 잘 쓰지 않는 페이지를 디스크로 보내지 않고, **RAM 안에서 압축한 채로 보관**한다. 디스크 I/O가 아예 없으니 플래시 수명도 안 닳고, 압축 해제도 빠르다.

iOS는 이 방식을 정식으로 채택했고, 안드로이드도 **zRAM** 형태로 일부 기기에서 사용한다. 자세한 건 10.7절에서 다시 다룬다.
