# 페이지 교체

## LRU(least recently used) 페이지 교체

가장 최근에 사용된 것을 미래에 사용될 가능성이 크다고 보기

→ 반대로 가장 오래 전에 사용된건 앞으로도 사용되지 않을 것이라고 간주됨

구현은 아래 두 가지 방법으로 가능

- 카운터: 각 페이지마다 마지막 사용 시간을 저장. 비교해서 더 작은 값(예전 시각)을 교체
    - 시간값 저장 오버헤드 발생
- 스택: 참조될 때마다 스택의 맨 앞에 넣음. 맨 뒤가 가장 오래전에 사용한 것
    - 이중 연결 리스트를 사용하여 오버헤드 큼
    - 소프트웨어나 마이크로 코드에서 사용
    
    *마이크로 코드: CPU 안에서 명령어를 처리하는 매우 로우한 레벨의 코드. 이중 연결 리스트 사용 방식때문에 오버헤드가 커서 하드웨어 수준이 로우 레벨 코드에서 활용해야 한다고 한 것. 
    

⇒ 두 방식 다 오버헤드가 커서 실무에 거의 사용되지 않음.

하드웨어에서 제공하는 **참조 비트**는 사용함. 페이지가 사용되면 1로 세팅. 사용되지 않은 페이지는 거를 수 있음

### 부가적 참조 비트 알고리즘

컨베이어벨트가 돌아가고 계속 동일 위치에 도장을 찍으면 몇 번 그 위치에 도장이 찍혔는지 알 수 있듯이,

참조 비트가 올라가면 shift 레지스터로 한 칸씩 옮겨 가면서 적재하여 어떤 페이지가 많이 사용되었는지 체크 가능

### 2차 기회 알고리즘

순환 큐를 이용해서 참조 비트를 하나씩 검사. 0이면 대체, 1이면 봐주고 0으로 세팅. 한바퀴를 다 돌면

사실상 FIFO

### 개선된 2차 기회 알고리즘

참조 비트와 변경 비트를 동시에 사용하여 4등급으로 나눔

 낮은 등급을 찾기 위해 큐를 여러번 검사하게 될 수 있음

⇒ MFU알고리즘은 LRU의 반대. 많이 참조된 것이 앞으로 참조될 가능성이 낮다

⇒ LRU, MFU 모두 알고리즘을 구현하는데 비용이 많이 들고 최적 페이지 정책을 따르지 못하기 때문에 사용되지 않음

## 페이지-버퍼링 알고리즘

가용 프레임 여러 개를 풀로 가지고 있다가 페이지 폴트가 발생하면 제공하는 것

방식 두 개가 있음

1. 기본 방식: victim 페이지를 고르기는 함. 근데 **victim이 디스크로 들어가는걸 기다리지 않고** 비동기적으로 free frame pool을 읽어서 빈공간을 우선 쓰게 한 다음에 나중에 victim이 디스크에 써지고나면 그 프레임은 free pool로 들어감
2. 확장 방식: 변경된 페이지 리스트(메모리에 올라온 후 변경된 페이지)를 모아뒀다가 디스크가 한가할 때 미리 디스크에 써두고 dirty bit를 0으로 변경. 페이지 폴트가 발생했을 때 즉시 victim을 디스크에 쓰는 것이 아니라 victim으로 선정될 가능성이 커짐 → 효율적
3. free frame pool이 원래 어떤 페이지였는지를 기억해뒀다가 나중에 필요할 때 바로 가져옴

# 프레임 할당

### 최소로 할당해야 할 프레임 수

한 명령어를 돌릴 때 여러 페이지가 필요할 수 있는데 페이지 폴트가 발생하면 이 명령어를 다시 수행해야 한다.

보통 컴퓨터 아키텍처에 의해 결정됨. 최대로 할당할 프레임수는 물리 메모리 양에 의해 정의됨

### 할당 알고리즘

- 균등 할당
- 비례 할당: 프로세스 크기 비율에 맞춰서.

⇒ 우선순위를 고려하려면 비례 할당을 사용하되 비율을 프로세스 크기가 아니라 우선순위대로 할당하거나 크기x우선순위 조합 사용

- 전역 교체: 어떤 프로세스의 프레임도 뺏어올 수 있음 → 다른 프로세스의 동작에 간섭하게 될 수 있음
- 지역 교체: 내 프로세스 안에서의 프레임만 뺏어올 수 있음 → 어떤 프로세스는 계속 실패할 수 있음

⇒ 전역 교체가 더 성능이 좋음

가용 프레임이 완전 0일 때 페이지 교체를 하지 않고 임계값 아래로 떨어지면 교체하는 방식도 있다.

임계값 아래로 떨어지면 커널이 페이지를 회수를 재개하고 최대 임계에 도달하면 페이지 회수를 중단한다.

- Non-Uniform Memory Access (NUMA): 스레드가 실행되는 CPU 근처의 메모리에 페이지를 배치하는 방식.

# Thrashing

필요한 프레임이 부족할 때 계속해서 페이지 폴트가 발생하는 현상. 실행보다 페이징에 더 많은 시간을 사용하면 스래싱이 발생했다고 함 → 심각한 성능 문제

방지하기 위해서는 각 프로세스가 필요로 하는 최소 프레임 개수를 알아야함

→ 실제로 사용하고 있는 프레임 수를 확인

- 지역성 모델: 프로세스가 실행될 때 항상 특정 지역의 메모리를 집중 참조하게 되는데, 이 페이지들의 집합을 의미한다.

즉 지역성 모델을 담을 정도로 충분하지 않으면 스래싱이 발생한다.

- 작업 집합 모델: 최근 활발하게 참조한 서로 다른 페이지의 집합을 작업 집합이라고 한다.

이걸 필요 프레임 수라고 보고 이게 충분하지 않으면 스래싱이 발생한다.

### 방지 방법

페이지 폴트율의 상한, 하한을 정해놓고, 상한을 넘으면 그 프로세스에 프레임을 더 할당하고 하한보다 낮으면 프레임을 눌이는 방식
