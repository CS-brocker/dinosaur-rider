# 운영체제 스터디 13주차 정리

## LRU 페이지 교체

\: 가장 오랜 기간 동안 사용되지 않은 페이지를 교체

### 페이지의 마지막 사용시간 저장 구현 방법

1. 계수기(counters)
    - 각 페이지 항목마다 사용 시간 필드를 넣고, CPU에 논리적인 시계나 계수기를 추가한다.
    - 메모리 참조 때마다 페이지 테이블을 탐색해야 하고 쓰기 작업이 필요하다.
2. 스택(stack)
    - 페이지가 참조될 때마다 페이지 번호를 스택 중간에서 제거하고, top에 놓는다. bottom에는 가장 오래 사용되지 않은 페이지가 놓이게 된다.
    - 보통 이중 연결 리스트로 구현
        - 최악의 경우 6개의 포인터 값을 바꿔야 한다.

          A - B - C - D - E 에서 C를 갱신해야 한다고 하면

            1. [B.next](http://B.next) = D
            2. D.prev = B
            3. C.prev = null
            4. [C.next](http://C.next) = A
            5. A.prev = C
            6. head = C
    - 갱신 할 때는 오버헤드가 크지만, 페이지를 교체해야 할 때는 탐색이 필요 없다.

→ 메모리 참조 때마다 갱신해줘야 하기 때문에 하드웨어 지원이 없이 소프트웨어로만 구현한다면 오버헤드가 너무 크다.

### 스택 알고리즘

\: LRU 교체와 같이 Belady 모순이 발생하지 않는 교체 알고리즘

- n + 1 프레임이 되더라도 n 프레임 집합을 포함하는 경우
- FIFO 에서는 페이지 테이블에 있는 상태로 다시 참조된다고 해서 순서를 다시 앞으로 바꿔주지 않기 때문에 Belady 모순이 발생하지만, LRU 에서는 참조될 때마다 그 페이지의 순서를 바꿔준다.

## LRU 근사 페이지 교체

\: LRU 페이지 교체를 위해선 하드웨어 지원이 필요한데 지원하지 않는 하드웨어가 많음. → 대부분의 시스템에 있는 reference bit 를 활용

### 참조 비트를 활용한 알고리즘 1

\: 배경 개념: 주기적으로 참조 비트를 기록해서 최근에 참조된 페이지를 간접적으로 확인한다.

<예시>

1. 8비트로 참조 비트 이력을 저장할 수 있는 별도 값을 추가한다.
2. 처음에는 00000000 로 설정된다.
3. 페이지가 참조 됐을 때 참조 비트가 1이 된다.
4. 일정 시간이 지날 때마다 오른쪽으로 쉬프트해주고, 참조 비트를 확인해서 최상위 비트에 반영한다. → 10000000
5. page fault 발생했을 때 가장 작은 걸 골라서 교체한다.

### 2차 기회 알고리즘

\: 주기적으로 참조 비트를 조회해 기록하지 않고, 페이지들의 참조 비트를 FIFO 큐처럼 줄 세워 놓았다가 페이지 교체가 필요할 때 각 페이지의 참조 비트를 확인해서 교체하는 방식이다.

- 순서대로 참조 비트를 확인하면서 참조 비트가 0인 것을 찾을 때까지 다음으로 넘어간다. 그리고 한 번 확인한 참조 비트는 0으로 바꿔준다.

<예시>

A(1)-B(1)-C(0)-D(1)

1. A 확인 → A(0)-B(1)-C(0)-D(1)
2. B 확인 → A(0)-B(0)-C(0)-D(1)
3. C 확인 → C 교체

```c
❓victim 페이지를 찾기 위해 검사하는 동안 참조가 다시 일어나면 무한히 순환하는 구조가 되지 않나?

- 한 번 확인한 후에 다시 참조가 됐다는 것은 그만큼 자주 참조가 되고 있다는 것이기 때문에 교체되지 않는 것이 맞다.
	-> 교체된다면 그 페이지가 금장 다시 참조되어야 하기 때문에 page fault가 또 일어날 것

- 그럼에도 무한히 victim page만 찾고 새로운 페이지가 참조되지 못하고 계속 기다린다면?
	-> 하드웨어적으로 메모리가 부족한 것
```

### 클록 알고리즘

\: 2차 기회 알고리즘을 순환 큐로 구현한 것

### 개선된 2차 기회 알고리즘

\: 클록 알고리즘과 동일하게 동작하되 참조 비트만 확인하는 것이 아니라 참조 비트와 변경 비트를 사용해서 등급을 나누고, 등급이 가장 낮은 것 중에 처음 확인한 페이지를 교체한다.

<등급>

(참조 비트, 변경 비트)

1. (0, 0): 최근에 사용도, 변경도 되지 않은 경우 → 교체하기 가장 좋은 페이지
2. (0, 1): 변경이 일어나서 디스크에 내용을 기록해야 함.
3. (1, 0): 곧 다시 사용될 가능성이 높음.
4. (1, 1): 사용도 되고, 변경도 일어나서 교체하기 적합하지 않음.

## 계수 기반 페이지 교체

### LFU(least frequently used) 알고리즘

\: 참조 횟수가 가장 작은 페이지를 교체하는 방법

- 문제점: 초기 단계에서는한 페이지를 집중적으로 많이 사용하지만, 그 후로 다시는 이 페이지를 사용하지 않는 경우 → 참조 횟수는 커졌는데 더이상 사용하지 않는 페이지가 메모리에 계속 무무른다.
- 해결 방법: 일정 시간마다 하나씩 오른쪽으로 shift 해서 지수적으로 그 영향력을 감소시킨다.

### MFU(most frequently used) 알고리즘

\: 가장 많이 참조된 페이지를 교체하는 방법

- 이미 많이 사용된 페이지는 필요한 작업이 거의 끝났다고 보는 것이다.

→ LFU, MFU는 잘 사용되지 않는다. (최적 페이지 교체 정책에 제대로 근사하지 못함. 지나가던 x가 봐도 그래보임.)

## 페이지-버퍼링 알고리즘

\: 페이지 교체가 필요할 때 디스크 I/O 지연을 줄이기 위해서 교체된 페이지를 바로 버리지 않고 잠시 버퍼에 보관하는 방식

### 변경된 페이지 리스트 확장

\: 디스크에 쓰는 시간을 기다리지 않도록 함.

<예시>

1. A 페이지가 올라와 있음.
2. B 페이지를 사용해야 하는데 페이지 폴트가 발생함.
3. A 페이지를 쫓아내야 함.
4. A 페이지가 변경됐다면 modified page list에 넣음.

   → modified page list가 아니라면 여기서 바로 A 페이지를 디스크에 씀.

5. B 를 읽어옴.
6. 나중에 백그라운드나 동작이 없을 때 modified page list에 저장된 페이지를 디스크에 씀.

### 원래 임자 페이지를 기억해놓는 확장

\: 가용 프레임 풀에 들어간 프레임이 마지막으로 어떤 페이지를 담고 있었는지를 기억해둔다.

<예시>

1. A 페이지가 프레임 5에 있었음.
2. A 가 교체되면서 페이지 테이블에서 A 를 invalid 로 변경함.
3. 프레임 5를 가용 프레임 풀에 넣고, 프레임 5의 원래 임자가 A 였다고 기록함.
4. A 가 다시 필요해졌을 때  프레임 5가 아직 다른 페이지로 덮어써지지 않았다면, 디스크에서 A 를 다시 읽지 않고 프레임 5에 남아 있던 A 를 그대로 사용함.

→ 물리 메모리에서 프레임을 비운다는 것이 그 안의 데이터를 즉시 지운다는 것은 아니라는 점을 활용한 것

## Raw disk

DB와 같이 자체적으로 버퍼를 구축하는 경우 OS의 버퍼랑 중복 적용되면서 오히려 성능이 안 좋아질 수 있음.

→ OS 에서 관리되는 버퍼를 사용하지 않고 디스크에 직접 접근할 수 있도록 제공됨.
