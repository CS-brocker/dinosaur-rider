# 운영체제 스터디 14주차 정리

## 10.7 메모리 압축

\: 물리 메모리가 부족할 때 페이지를 바로 디스크로 스왑하지 않고, 메모리 안에서 압축된 형태로 보관하는 기법

<목적>

디스크 I/O를 줄이는 것 → 디스크 페이지를 내보내고 읽어오는 것은 매우 느리기 때문에 CPU 시간을 조금 써서 페이지를 압축하고 RAM 안에 보관하면 전체 성능이 좋아질 수 있음.

<동작 흐름>

1. 사용 가능한 프레임이 부족해진다.
2. 교체 대상 페이지를 고른다.
3. 그 페이지를 디스크에 쓰는 대신 압축한다.
4. 압축된 페이지를 메모리의 압축 영역을 저장한다.
5. 나중에 해당 페이지가 필요하면 압축을 풀어 다시 프레임에 올린다.

- 장점: 디스크 스왑보다 빠르고, 실제 사용 가능한 메모리 용량이 늘어난 것처럼 보인다.
- 단점: 압축/해제에 CPU가 필요하고, 데이터에 따라 압축률이 달라진다.

## 10.8 커널 메모리 할당

일반 사용자 프로세스의 메모리와 달리 커널 메모리는 특별한 조건이 많다.

커널은 자료구조, I/O 버퍼, 장치 드라이버용 메모리 등을 할당해야 한다. 특히 일부 장치는 DMA 때문에 물리적으로 연속된 메모리를 요구할 수 있다.

또한 커널 객체들은 크기가 작고 자주 생성/삭제되므로 일반 페이지 단위 할당만으로 비효율적이다.

### 메모리 할당 방식

1. Buddy System

   : 메모리를 2의거듭제곱 크기 블록으로 관리한다.

- 장점: 병합이 쉽고 빠르다.
- 단점: 요청 크기가 2의 거듭제곱과 딱 맞지 않으면 재부 단편화가 생긴다

1. Slab Allocation

   : 커널에서 자주 쓰는 객체들을 위한 캐시 기반 할당 방식

<예시>

디스크립터, 파일 객체, 세마포어 같은 커널 객체는 반복해서 생성된다.
→ Slab allocator는 이런 객체별로 캐시를 만들고, 미리 할당된 slab 안에서 객체를 빠르게 꺼내 쓴다.

<장점>

- 객체 생성/삭제가 빠르다.
- 같은 타입 객체를 재사용하므로 초기화 비용이 줄어든다.
- 내부 단편화를 줄일 수 있다.
- 커널 자료구조 할당에 적합하다.

## 10.9 기타 고려 사항

가상 메모리 시스템을 설계할 때는 페이지 교체 알고리즘 외에 실제 성능에는 여러 요소의 영향을 받는다.

### Prepaging

\: 페이지 폴트가 발생할 때마다 하나씩 가져오는 대신 앞으로 필요할 것 같은 페이지들을 미리 가져오는 방식

- 프로세스가 시작되거나 스왑 아웃되었다가 다시 들어올 때 많은 페이지 폴트가 연속적으로 발생할 수 있다. 이 때 관련 페이지들을 미리 가져오면 초기 페이지 폴트 수를 줄일 수 있다.
- But 예측이 틀리면 쓰지도 않은 페이지를 메모리에 올리는 낭비가 된다.

### Page Size

<작은 페이지>

- 장점
    - 내부 단편화가 적다.
    - 필요한 데이터만 메모리에 올리 수 있다.
- 단점
    - 페이지 테이블이 커진다.
    - TLB 효율이 떨어질 수 있다.
    - 페이지 폴트 수가 증가할 수 있다.

<큰 페이지>

- 장점
    - 페이지 테이블 크기가 작아진다.
    - TLB reach가 커진다.
    - 디스크 I/O 관점에서 효율적일 수 있다.
- 단점
    - 내부 단편화가 커질 수 있다.
    - 실제로 쓰지 않는 데이터까지 메모리에 올라올 수 있다.

→ 현대 시스템은 보통 기본 페이지 크기와 함께 huge page 같은 큰 페이지도 지원한다.

### TLB Reach

\: TLb가 한 번에 커버할 수 있는 메모리 범위

```c
TLB reach = TLB 엔트리 수 x 페이지 크기
```

- TLB reach가 프로세스의 working set보다 작으면 TLB miss가 자주 발생해 성능이 떨어진다.
  → TLB 엔트리 수를 늘리거나, 페이지 크기를 키우거나, huge page를 사용할 수 있다.

### Inverted Page Table

일반 페이지 테이블은 프로세스마다 가상 페이지 번호를 기준으로 엔트리를 가진다. 프로세스 주소 공간이 커질 수록 페이지 테이블도 커진다.
→ Inverted page table은 반대로 물리 프레임마다 하나의 엔트리를 둔다. 그래서 메모리 사용량을 크게 줄일 수 있다.

- 단점
    - 가상 주소를 물리 주소로 변환할 때 검색이 더 복잡해질 수 있다. → 보통 해시 테이블을 함께 사용한다.
    - 공유 메모리 구현도 일반 페이지 테이블보다 까다로울 수 있다.

### Program Structure

프로그램의 메모리 접근 패턴도 페이지 폴트에 영향을 준다.

<예시>

2차원 배열이 행 우선 순서로 저장되는 언어에서는 행 단위로 순차 접근하는 것이 좋다.
열 단위로 건너뛰며 접근하면 페이지를 자주 바꾸게 되어 페이지 폴트가 증가할 수 있다.

→ 같은 알고리즘이라도 데이터 접근 순서에 따라 가상 메모리 성능이 크게 달라진다.

### I/O Interlock and Page Locking

I/O 작업 중인 페이지는 디스크나 장치가 직접 접근할 수 있다. 이때 운영체제가 해당 페이지를 교체해 버리면 문제가 발생한다.

→ I/O에 사용 중인 페이지는 메모리를 고정한다.

## 10.10 운영체제의 예

### Windows

- demand paging을 사용하며, 프로세스별 working set을 관리한다. 각 프로세스는 일정한 수의 페이지를 메모리에 유지하고, 메모리가 부족하면 working set을 조정하거나 페이지를 교체한다.
- free list, zeroed list, modified list, standby list 같은 여러 페이지 목록을 관리해 페이지 상태를 구분한다. modified page는 디스크에 기록되어야 재사용될 수 있다.
- 최근 Windows 계열은 메모리 압축도 사용하여 디스크 스왑을 줄인다.

### Solaris

- free memory 양에 따라 페이지 회수 정책을 조절한다. 사용 가능한 메모리가 충분하면 별다른 회수를 하지 않지만, 특정 임계값 아래로 떨어지면 page scanner가 동작해 사용 빈도가 낮은 페이지를 회수한다.
- lotsfree, desfree, minfree 같은 기준값을 두어 메모리 압박 정도에 따라 스캔 강도를 조절한다.

## Linux

- demand paging, copy-on-write, page cache, buddy system, slab 계열 allocator 등을 사용한다.
- 물리 메모리는 zone으로 나뉘어 관리될 수 있고, 페이지 교체는 active/inactive 리스트 같은 구조를 통해 이루어진다. 자주 쓰이는 페이지는 active 쪽에 남고, 덜 쓰이는 페이지는 inactive로 이동해 교체 후보가 된다.
- 커널 메모리 할당에 buddy system과 slab 계열 allocator를 사용한다.
