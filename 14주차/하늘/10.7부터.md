### 10.7 메모리 압축

공간이 부족할 때, 수정된 프레임을 디스크로 페이징 아웃하지 않고
여러 프레임을 하나의 프레임으로 압축하는 방식.

나중에 접근하려고 하면 페이지 폴트를 발생시키고 압축을 푸는 과정이 필요하지만 그래도 스왑했다가 나중에 다시 가져오는 것보다 빠름

iOS, Android, Window 10, macOS 모두 지원

### 10.8 커널 메모리 할당

1. **Buddy System** 
    
    2의 거듭제곱 크기로만 나누어서 할당받는 것.
    
    빠르지만 내부 단편화 발생함
    
    1. 100KB 요청
        
        ```swift
        1024
        └512
        └256
        └128 ← 사용
        ```
        
    2. 반납
        
        ```swift
        128 + 128
        ↓
        256
        
        256 + 256
        ↓
        512
        
        512 + 512
        ↓
        1024
        ```
        
2. **Slab Allocation**
    
    미리 자주 사용하는 객체를 위한 공간을 확보해둠
    빠르고 내부 단편화를 줄일 수 있고 한번 초기화된 객체가 갖고 있는 공간(캐시)을 나중에 재사용한다
    
    사용 후 해당 메모리를 가용한 것으로 완전 비우지 않고 free로 표시하여 나중에 그 공간을 사용한다. 
    
    ```swift
    // Parial
    Slab A
    [used]
    [free]
    [used]
    [free]
    
    // Empty
    Slab B
    [free]
    [free]
    [free]
    [free]
    => 이 때는 partial 먼저 사용함. Slab B는 비워둬야 나중에 한 번에 반납하기 때문..
    ```
    

### 10.9 기타 고려 사항

1. 프리페이징 (Prepaging)
    - 페이지 가져올 때 사용될 것 같은 다른 페이지들도 미리 가져옴
2. 페이지 크기
    - 페이지 크기가 작으면 내부 단편화가 감소하지만 페이지 수가 증가하므로 페이지 테이블이 증가하고 페이지 폴트가 날 확률이 높아진다.
    - 페이지의 크기를 늘리면 I/O 시간이 증가하니까 페이지 크기가 작은게 좋을 것 같지만, 각 페이지당 지연시간과 탐색시간이 들어가기 때문에 실제로는 큰게 낫다.
    - 내부 단편화나 지역성을 생각하면 작은게 낫지만 페이지 테이블 크기나 입출력 효율을 보면 큰게 낫다.
3. TLB Reach
    - TLB가 커버 가능한 메모리 크기 (TLB의 엔트리수 x 페이지 크기)
    - 페이지 크기를 키우거나 엔트리수를 늘릴 수 있겠으나 TLB에 사용되는 연관메모리는 비싸고 전력도 많이 소모된다
4. 역페이지 테이블
    - 기존 페이지테이블은 가상페이지를 기준으로 물리프레임을 맵핑하지만 역으로는 물리 프레임을 기준으로 관리한다. 페이지 테이블을 줄일 수 있다.
5. 프로그램 구조
    - 프로그래밍 할 때 행 대신 열을 먼저 채우도록 지역성을 고려하여 구현하자
6. I/O 상호 잠금과 페이지 잠금
    - 디스크가 메모리 버퍼에 데이터를 쓰고 있는 중에 OS가 해당 페이지를 회수하면 문제 발생하니, 교체 금지할 수 있도록 잠금 비트를 활용하여 lock을 건다.
    - 이렇게 페이지를 고정하는 것을 pinning이라고 하고 대부분의 운영체제는 자신의 영역을 메모리에 고정할 수 있는 시스템 콜이 있다
    - 이외에는 어떤 블록을 쓰고 싶으면 시스템 메모리로 복사하고 거기서 디스크에 쓰기 작업을 하는 방법도 있다.

### 10.10 운영체제 예

소개하는 Linux, Windows, Solaris 모두 최근 사용한 페이지는 살리고 오래 안쓴 페이지는 버린다는 점은 같다. 

- Linux
    - LRU, 슬랙 할당 사용
    - active_list: 사용 중인 것으로 간주하는 페이지
    - inactive_list: 최근 참조되지 않았고 회수할 수 있는 페이지
    - accessed 비트
    - 할당되면 accessed 비트가 설정되고 active_list 끝에 추가되고 참조될 때마다 active_list의 리스트의 뒤로 이동하고 비트가 계속 설정된다.
    - 주기적으로 비트가 리셋되는데, 시간이 지남에 따라 최근에 참조된 적 없는 페이지가 맨 앞에 있게 되고 이 때 inactive_list의 맨 뒤로 이동한다.
